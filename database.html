Introduction to Databases

What is a Database?

A database is an organized collection of data that can be easily accessed, managed, and updated. Databases are used to store information like user data, products, orders, posts, and more in web applications and software systems.

Why Use a Database?

  • Store large amounts of structured data efficiently.
  • Retrieve, filter, and sort data quickly using queries.
  • Maintain relationships between different sets of data (e.g., users and orders).
  • Ensure data integrity, security, and backups.
  • Allow multiple users and applications to access data simultaneously.

Types of Databases

  • Relational Databases (RDBMS): Use tables, rows, and columns. Examples: MySQL, PostgreSQL, SQLite.
  • NoSQL Databases: Use flexible formats like JSON. Examples: MongoDB, Firebase, CouchDB.
  • In-Memory Databases: Store data in RAM for speed. Example: Redis.
  • Cloud Databases: Hosted databases like Amazon RDS, Google Firebase.

Basic Database Terms

  • Table: A collection of related data in rows and columns.
  • Row (Record): A single data item or entry in a table.
  • Column (Field): A single type of data (like name, email).
  • Primary Key: A unique identifier for each record in a table.
  • Foreign Key: A field that links to another table’s primary key.

Example (SQL - Relational Database)

-- Create a table
CREATE TABLE users (
  id INT PRIMARY KEY,
  name VARCHAR(100),
  email VARCHAR(100)
);

-- Insert data
INSERT INTO users (id, name, email)
VALUES (1, 'Alice', 'alice@example.com');

-- Query data
SELECT * FROM users;

Example (NoSQL - MongoDB)

// Insert a document into a collection
db.users.insertOne({
  name: "Bob",
  email: "bob@example.com"
});

// Find documents
db.users.find();

How Databases Work in Web Apps

Databases connect to your application using a backend language (e.g., Node.js, PHP, Python). Queries are sent from your app to the database to store, update, or retrieve data.

Example flow:

  1. User submits a form on the website.
  2. JavaScript sends the data to the backend (e.g., Node.js).
  3. The backend executes a query and saves it to the database.
  4. The database responds with success or failure.

Keys in DBMS

Keys are crucial in relational databases as they uniquely identify records and help maintain data integrity. Here are the main types of keys:

1. Primary Key

A Primary Key uniquely identifies each record in a table. It cannot contain NULL values and must be unique.

Syntax:

CREATE TABLE Students ( ID INT PRIMARY KEY, Name VARCHAR(100) );

Example: ID = 101 is a unique student identifier.

2. Foreign Key

A Foreign Key is a field that links one table to another. It refers to the primary key in another table.

Syntax:

CREATE TABLE Orders ( OrderID INT, CustomerID INT, FOREIGN KEY (CustomerID) REFERENCES Customers(CustomerID) );

Example: Orders table references CustomerID from Customers table.

3. Candidate Key

Candidate Keys are all possible keys that can uniquely identify rows. One is chosen as the primary key.

-- Candidate Keys: Email, RollNumber CREATE TABLE Students ( RollNumber INT, Email VARCHAR(100), Name VARCHAR(100), PRIMARY KEY (RollNumber) );

Example: Both Email and RollNumber can be candidate keys.

4. Composite Key

A Composite Key is a combination of two or more columns used to uniquely identify a record.

CREATE TABLE CourseEnrollments ( StudentID INT, CourseID INT, PRIMARY KEY (StudentID, CourseID) );

Example: One student can take many courses. Together, StudentID + CourseID make a unique key.

5. Alternate Key

Alternate Keys are candidate keys not selected as the primary key.

Example: If RollNumber is primary key, then Email is an alternate key.

6. Super Key

A Super Key is any combination of attributes that uniquely identify a row. It may include extra unnecessary fields.

Example: (ID, Name) is a super key, but ID alone is sufficient and preferred.

Normalization in DBMS

Normalization is the process of organizing data in a database to reduce redundancy and improve data integrity. It involves dividing large tables into smaller ones and defining relationships between them.

🔍 Why Normalize?

  • Eliminate redundant data
  • Ensure data dependencies make sense
  • Make the database easier to maintain
  • Prevent anomalies (update, insert, delete)

1NF (First Normal Form)

Each column must contain atomic (indivisible) values, and each record must be unique.

Unnormalized Table:

StudentID | Name | Courses 1 | Alice | Math, Science 2 | Bob | Math

1NF Table: Break multivalued fields into separate rows

StudentID | Name | Course 1 | Alice | Math 1 | Alice | Science 2 | Bob | Math

2NF (Second Normal Form)

Must be in 1NF and all non-key attributes must depend on the whole primary key.

Problem: In a composite key table, some columns depend only on part of the key.

Table: CourseEnrollment StudentID | CourseID | StudentName | CourseName

2NF Solution: Separate into two tables

-- Students Table StudentID | StudentName -- Courses Table CourseID | CourseName -- Enrollments Table StudentID | CourseID

3NF (Third Normal Form)

Must be in 2NF, and all attributes must depend *only* on the primary key, not on other non-key attributes.

Example: If DepartmentName depends on DepartmentID, and DepartmentID depends on EmployeeID, there's a transitive dependency.

-- Before: EmployeeID | EmployeeName | DepartmentID | DepartmentName -- After 3NF: Employees: EmployeeID | EmployeeName | DepartmentID Departments: DepartmentID | DepartmentName

BCNF (Boyce-Codd Normal Form)

A stronger version of 3NF. A table is in BCNF if every determinant is a candidate key.

Problem: Multiple candidate keys but not all are primary.

Professor | Subject | Department (Professor + Subject) → Department Subject → Department

Solution: Split into two tables where Subject → Department is also a key.

4NF (Fourth Normal Form)

Must be in BCNF and have no multi-valued dependencies (i.e., multiple independent facts about the same key).

StudentID | Language | Hobby -- Here, Language and Hobby are independent. -- Break into two tables: StudentLanguages: StudentID | Language StudentHobbies: StudentID | Hobby

5NF (Fifth Normal Form)

Deals with reconstructing information from smaller pieces without redundancy. Used in very complex databases with multi-join dependencies.

Rarely used in practice but ensures full normalization without join dependency issues.

📝 Summary Table

Normal Form Condition
1NF Atomic values (no repeating groups)
2NF 1NF + No partial dependency on composite key
3NF 2NF + No transitive dependency
BCNF Every determinant is a candidate key
4NF BCNF + No multi-valued dependencies
5NF No join dependencies

Joins in SQL

Joins are used to combine rows from two or more tables based on a related column between them.

1. INNER JOIN

Returns records that have matching values in both tables.

SELECT students.name, courses.title FROM students INNER JOIN enrollments ON students.id = enrollments.student_id INNER JOIN courses ON enrollments.course_id = courses.id;

Only students who are enrolled in a course will be shown.

2. LEFT JOIN (LEFT OUTER JOIN)

Returns all records from the left table, and the matched records from the right table. Returns NULL if there is no match.

SELECT students.name, courses.title FROM students LEFT JOIN enrollments ON students.id = enrollments.student_id LEFT JOIN courses ON enrollments.course_id = courses.id;

All students will be listed, even those not enrolled in any course.

3. RIGHT JOIN (RIGHT OUTER JOIN)

Returns all records from the right table and matched records from the left table.

SELECT students.name, courses.title FROM students RIGHT JOIN enrollments ON students.id = enrollments.student_id RIGHT JOIN courses ON enrollments.course_id = courses.id;

4. FULL JOIN (FULL OUTER JOIN)

Returns all records when there is a match in either left or right table. Shows NULL where there's no match.

SELECT students.name, courses.title FROM students FULL OUTER JOIN enrollments ON students.id = enrollments.student_id FULL OUTER JOIN courses ON enrollments.course_id = courses.id;

5. CROSS JOIN

Returns the Cartesian product of both tables (every combination of rows).

SELECT * FROM students CROSS JOIN courses;

Relational Algebra

Relational Algebra is a procedural query language that operates on relations (tables) and returns new relations. It's used to describe the logic behind SQL operations.

Core Relational Algebra Operations

  • σ (Selection): Selects rows that satisfy a condition.
    σ age > 25 (Employees)
  • π (Projection): Selects specific columns.
    π name, age (Employees)
  • ⋈ (Join): Combines two relations based on a condition.
    Employees ⋈ Employees.department_id = Departments.id Departments
  • × (Cartesian Product): Combines all rows from two relations.
    Employees × Departments
  • ∪ (Union): Combines tuples from two relations (must be compatible).
    Managers ∪ Developers
  • − (Difference): Finds tuples in one relation but not in another.
    Employees − Managers
  • ρ (Rename): Renames the relation or attributes.
    ρ Emp(Name, Age) (Employees)

📘 Example Query in Relational Algebra

Get names of employees who work in the IT department:

π name (σ department = 'IT' (Employees))

SQL Constraints

Constraints are rules enforced on data columns to ensure data integrity and accuracy in the database.

1. NOT NULL

Ensures that a column cannot have a NULL value.

CREATE TABLE users ( id INT, username VARCHAR(100) NOT NULL );

2. UNIQUE

Ensures all values in a column are different (no duplicates).

CREATE TABLE users ( email VARCHAR(255) UNIQUE );

3. PRIMARY KEY

Uniquely identifies each record in a table. It is a combination of NOT NULL and UNIQUE.

CREATE TABLE users ( id INT PRIMARY KEY, name VARCHAR(100) );

4. FOREIGN KEY

Establishes a relationship between two tables. It ensures the value in one table exists in another.

CREATE TABLE orders ( id INT, user_id INT, FOREIGN KEY (user_id) REFERENCES users(id) );

5. CHECK

Ensures that all values in a column satisfy a specific condition.

CREATE TABLE products ( price DECIMAL(10, 2), CHECK (price > 0) );

6. DEFAULT

Sets a default value for a column when no value is specified.

CREATE TABLE users ( is_active BOOLEAN DEFAULT true );

7. INDEX

Improves the speed of data retrieval operations on a table.

CREATE INDEX idx_username ON users(username);

📌 Summary of Common Constraints

  • NOT NULL – Value must be provided.
  • UNIQUE – No duplicates allowed.
  • PRIMARY KEY – Unique + Not Null (identity of the row).
  • FOREIGN KEY – Relates two tables.
  • CHECK – Validates data with a condition.
  • DEFAULT – Auto-inserts value when none is given.
  • INDEX – Optimizes lookups, not a constraint per se, but supports performance.

Set Operations in SQL & Relational Algebra

Set operations are used to combine the results of two or more SQL queries. These operations follow set theory and require the queries to have the same number of columns and compatible data types.

1. UNION

Definition: Combines results from two queries and removes duplicates.

SQL Syntax:

SELECT column1 FROM table1 UNION SELECT column1 FROM table2;

Example:

SELECT name FROM developers UNION SELECT name FROM testers;

Returns all unique names from both tables.

2. UNION ALL

Definition: Combines results from two queries including duplicates.

SQL Syntax:

SELECT column1 FROM table1 UNION ALL SELECT column1 FROM table2;

Example:

SELECT name FROM developers UNION ALL SELECT name FROM testers;

Returns all names including duplicates from both tables.

3. INTERSECT

Definition: Returns only the common records from both queries.

SQL Syntax (PostgreSQL, Oracle):

SELECT column1 FROM table1 INTERSECT SELECT column1 FROM table2;

Example:

SELECT name FROM developers INTERSECT SELECT name FROM testers;

Returns names that exist in both tables.

4. EXCEPT / MINUS

Definition: Returns records from the first query that are not in the second.

SQL Syntax (PostgreSQL / Oracle):

SELECT column1 FROM table1 EXCEPT -- use MINUS in Oracle SELECT column1 FROM table2;

Example:

SELECT name FROM developers EXCEPT SELECT name FROM testers;

Returns names in developers that are not in testers.

Relational Algebra Equivalent

Set operations are also part of relational algebra, which operates on sets (relations) and returns a new relation.

  • R ∪ S → UNION
  • R ∩ S → INTERSECT
  • R - S → EXCEPT / MINUS
  • R × S → Cartesian Product

Union Compatibility

For set operations to work, the following must be true:

  • Same number of columns
  • Same data types or compatible data types

Use Cases

  • UNION: Get unique users from two departments
  • UNION ALL: Combine logs from multiple systems
  • INTERSECT: Find users common to two systems
  • EXCEPT: Users who left a service

SQL Queries & Nested Queries

SQL queries are commands used to retrieve or manipulate data stored in relational databases. Nested queries, also known as subqueries, are queries written inside another query.

1. Basic SQL Queries

  • SELECT: Retrieve data from one or more tables.
  • INSERT: Add new data into a table.
  • UPDATE: Modify existing data.
  • DELETE: Remove records from a table.
-- SELECT example SELECT name, age FROM users WHERE age > 21; -- INSERT example INSERT INTO users (name, age) VALUES ('Alice', 25); -- UPDATE example UPDATE users SET age = 26 WHERE name = 'Alice'; -- DELETE example DELETE FROM users WHERE name = 'Alice';

2. Subqueries (Nested Queries)

A subquery is a query within another SQL query. It is used to perform intermediate results that the main query depends on.

  • Can be used in SELECT, FROM, or WHERE clauses.
  • Can return single or multiple values.

3. Subquery in WHERE Clause

Find all users who are older than the average age.

SELECT name, age FROM users WHERE age > (SELECT AVG(age) FROM users);

4. Subquery in FROM Clause

Use a subquery to create a temporary result set.

SELECT department, AVG(salary) AS avg_salary FROM ( SELECT department, salary FROM employees ) AS temp GROUP BY department;

5. Correlated Subquery

A correlated subquery refers to a column in the outer query. It's executed once for each row in the outer query.

SELECT name FROM employees e1 WHERE salary > ( SELECT AVG(salary) FROM employees e2 WHERE e1.department = e2.department );

Explanation:

  • This finds employees whose salary is above the average salary of their own department.

6. Subquery in SELECT Clause

Shows how many orders each user has placed.

SELECT name, (SELECT COUNT(*) FROM orders WHERE orders.user_id = users.id) AS order_count FROM users;

Common Use Cases for Subqueries

  • Filtering data with conditional logic
  • Comparing with aggregated values (MAX, AVG, COUNT)
  • Building temporary result sets
  • Validating constraints or duplicates

Key Notes

  • Subqueries must be enclosed in parentheses ().
  • They can return scalar values (one value), rows, or even tables.
  • Correlated subqueries are slower due to multiple executions.

SQL Triggers

A trigger is a database object that automatically executes a predefined action in response to certain events on a table or view. Triggers help enforce business rules, log changes, or automate system tasks.

1. What is a Trigger?

  • Automatically fires when an INSERT, UPDATE, or DELETE operation occurs.
  • Can be defined to fire BEFORE or AFTER the event.
  • Used for auditing, validation, or enforcing rules.

2. Trigger Syntax (MySQL Example)

CREATE TRIGGER trigger_name {BEFORE | AFTER} {INSERT | UPDATE | DELETE} ON table_name FOR EACH ROW BEGIN -- SQL statements to execute END;

3. BEFORE INSERT Trigger

Automatically sets the created date before inserting a new user.

CREATE TRIGGER before_user_insert BEFORE INSERT ON users FOR EACH ROW BEGIN SET NEW.created_at = NOW(); END;

4. AFTER UPDATE Trigger

Logs old and new salary after an employee's salary is updated.

CREATE TRIGGER log_salary_update AFTER UPDATE ON employees FOR EACH ROW BEGIN INSERT INTO salary_log (emp_id, old_salary, new_salary, changed_at) VALUES (OLD.id, OLD.salary, NEW.salary, NOW()); END;

5. AFTER DELETE Trigger

Keeps a record of deleted users in an archive table.

CREATE TRIGGER archive_user AFTER DELETE ON users FOR EACH ROW BEGIN INSERT INTO deleted_users (id, name, deleted_at) VALUES (OLD.id, OLD.name, NOW()); END;

6. Common Use Cases for Triggers

  • Audit changes (track old and new values).
  • Auto-update timestamps (e.g., updated_at).
  • Maintain history/log tables.
  • Prevent invalid data entries.
  • Synchronize tables automatically.

7. Key Notes

  • NEW refers to new data being inserted/updated.
  • OLD refers to the existing data before update/delete.
  • Triggers run automatically — no need to call them explicitly.
  • Be cautious: triggers can cause performance issues or circular logic if not handled properly.

1. Data Definition Language (DDL)

DDL commands are used to define and modify the structure of database objects like tables, indexes, and schemas.

CREATE TABLE

Purpose: Creates a new table in the database.

Syntax:

CREATE TABLE table_name ( column1 datatype constraints, column2 datatype constraints, ... );

Example:

CREATE TABLE users ( id INT PRIMARY KEY, name VARCHAR(50) NOT NULL, age INT );

This creates a table named "users" with columns for ID, name, and age.

ALTER TABLE

Purpose: Modifies an existing table (e.g., adds, deletes, or modifies columns).

Syntax:

ALTER TABLE table_name ADD column_name datatype;

Example:

ALTER TABLE users ADD email VARCHAR(100);

This adds a new column "email" to the "users" table.

DROP TABLE

Purpose: Deletes an existing table and all its data.

Syntax:

DROP TABLE table_name;

Example:

DROP TABLE users;

This deletes the "users" table.

2. Data Manipulation Language (DML)

DML commands are used to manipulate data within database tables.

INSERT

Purpose: Adds new rows of data to a table.

Syntax:

INSERT INTO table_name (column1, column2, ...) VALUES (value1, value2, ...);

Example:

INSERT INTO users (id, name, age) VALUES (1, 'John', 30);

This inserts a new row into the "users" table.

UPDATE

Purpose: Modifies existing data in a table.

Syntax:

UPDATE table_name SET column1 = value1, column2 = value2, ... WHERE condition;

Example:

UPDATE users SET age = 31 WHERE id = 1;

This updates the age of the user with ID 1.

DELETE

Purpose: Removes rows from a table.

Syntax:

DELETE FROM table_name WHERE condition;

Example:

DELETE FROM users WHERE id = 1;

This deletes the user with ID 1 from the "users" table.

SELECT

Purpose: Retrieves data from a table.

Syntax:

SELECT column1, column2, ... FROM table_name WHERE condition;

Example:

SELECT name, age FROM users WHERE age > 25;

This retrieves the names and ages of users older than 25.

3. Data Query Language (DQL)

DQL commands are used to query and retrieve data from the database.

SELECT with WHERE

Purpose: Filters rows based on a condition.

Syntax:

SELECT column1, column2, ... FROM table_name WHERE condition;

Example:

SELECT * FROM users WHERE age > 25;

This retrieves all users older than 25.

ORDER BY

Purpose: Sorts the result set in ascending or descending order.

Syntax:

SELECT column1, column2, ... FROM table_name ORDER BY column1 ASC|DESC;

Example:

SELECT * FROM users ORDER BY age DESC;

This retrieves all users sorted by age in descending order.

GROUP BY

Purpose: Groups rows that have the same values into summary rows.

Syntax:

SELECT column1, COUNT(*) FROM table_name GROUP BY column1;

Example:

SELECT age, COUNT(*) FROM users GROUP BY age;

This groups users by age and counts the number of users in each age group.

HAVING

Purpose: Filters groups based on a condition.

Syntax:

SELECT column1, COUNT(*) FROM table_name GROUP BY column1 HAVING condition;

Example:

SELECT age, COUNT(*) FROM users GROUP BY age HAVING COUNT(*) > 1;

This retrieves age groups with more than one user.

4. Data Control Language (DCL)

DCL commands are used to control access to the database.

GRANT

Purpose: Gives specific privileges to users or roles.

Syntax:

GRANT privilege_name ON object_name TO user_name;

Example:

GRANT SELECT, INSERT ON users TO 'user1';

This grants SELECT and INSERT permissions on the "users" table to 'user1'.

REVOKE

Purpose: Removes specific privileges from users or roles.

Syntax:

REVOKE privilege_name ON object_name FROM user_name;

Example:

REVOKE INSERT ON users FROM 'user1';

This revokes INSERT permission on the "users" table from 'user1'.

5. Transaction Control Language (TCL)

TCL commands are used to manage transactions in the database.

COMMIT

Purpose: Saves all changes made during the current transaction.

Syntax:

COMMIT;

Example:

COMMIT;

This saves all changes made in the current transaction.

ROLLBACK

Purpose: Undoes all changes made during the current transaction.

Syntax:

ROLLBACK;

Example:

ROLLBACK;

This undoes all changes made in the current transaction.

SAVEPOINT

Purpose: Sets a savepoint within a transaction to which you can later roll back.

Syntax:

SAVEPOINT savepoint_name;

Example:

SAVEPOINT savepoint1;

This creates a savepoint named "savepoint1".

6. NoSQL Databases & Query Examples

NoSQL databases provide flexible data storage and retrieval methods, categorized into different types. Unlike traditional relational databases, NoSQL databases offer high scalability, flexibility, and speed for handling unstructured and semi-structured data.

Key-Value Stores (Redis, DynamoDB)

Purpose: Store data as key-value pairs. These databases are highly efficient for caching, session management, and real-time analytics.

Example (Redis):

SET user:1001 "{\"name\": \"John\", \"age\": 30}"

Retrieving the value:

GET user:1001

Example (DynamoDB Query):

{ "TableName": "Users", "Key": { "id": { "S": "1001" } } }

DynamoDB is fully managed and supports automatic scaling.

Document Stores (MongoDB, CouchDB)

Purpose: Store data in JSON-like documents, making it highly flexible for handling hierarchical data.

Example (MongoDB Insert):

db.users.insertOne({ name: "Alice", age: 25, email: "alice@example.com" })

Example (MongoDB Query):

db.users.find({ age: { $gt: 25 } })

MongoDB allows schema-less storage, making it easy to evolve applications without complex migrations.

Column-Family Stores (Cassandra, HBase)

Purpose: Store data in column-oriented structures, allowing fast read and write operations for large-scale distributed systems.

Example (Cassandra Create Table):

CREATE TABLE users (id UUID PRIMARY KEY, name TEXT, age INT);

Example (Cassandra Query):

SELECT * FROM users WHERE age > 30;

Column-family stores provide high availability and partition tolerance.

Graph Databases (Neo4j, ArangoDB)

Purpose: Store and query relationships between entities, making them ideal for social networks, fraud detection, and recommendation engines.

Example (Neo4j Create Node):

CREATE (a:Person {name: "David", age: 40})

Example (Neo4j Query):

MATCH (p:Person) WHERE p.age > 30 RETURN p.name, p.age;

Graph databases excel at handling complex connected data with efficient traversal queries.

5. Database Normalization (SQL Databases)

Database normalization is a process used to reduce redundancy and dependency by organizing fields and table relationships. It is divided into several normal forms:

1st Normal Form (1NF)

Definition: A table is in 1NF if it contains only atomic (indivisible) values and each column contains values of a single type.

Example:

Non-1NF Table: OrderID | Customer | Products ---------|-----------|---------------- 1 | Alice | Apple, Banana 2 | Bob | Orange, Grape 1NF Table: OrderID | Customer | Product ---------|-----------|-------- 1 | Alice | Apple 1 | Alice | Banana 2 | Bob | Orange 2 | Bob | Grape

In 1NF, multi-valued attributes are removed by creating separate rows.

2nd Normal Form (2NF)

Definition: A table is in 2NF if it is in 1NF and all non-key attributes are fully dependent on the primary key (i.e., no partial dependencies).

Example:

Non-2NF Table: OrderID | Product | SupplierID | SupplierName ---------|---------|------------|------------- 1 | Apple | S1 | FreshFarms 1 | Banana | S2 | OrganicFoods 2NF Tables: Orders Table: OrderID | Product | SupplierID ---------|---------|------------ 1 | Apple | S1 1 | Banana | S2 Suppliers Table: SupplierID | SupplierName ----------|-------------- S1 | FreshFarms S2 | OrganicFoods

In 2NF, supplier information is moved to a separate table to remove partial dependencies.

3rd Normal Form (3NF)

Definition: A table is in 3NF if it is in 2NF and all non-key attributes are only dependent on the primary key (i.e., no transitive dependencies).

Example:

Non-3NF Table: OrderID | CustomerID | CustomerName | CustomerCity ---------|-----------|--------------|------------- 1 | C1 | Alice | New York 2 | C2 | Bob | Los Angeles 3NF Tables: Orders Table: OrderID | CustomerID ---------|----------- 1 | C1 2 | C2 Customers Table: CustomerID | CustomerName | CustomerCity ----------|--------------|------------- C1 | Alice | New York C2 | Bob | Los Angeles

In 3NF, customer details are stored in a separate table to remove transitive dependencies.

5. Database Normalization (SQL Databases)

Database normalization is the process of organizing a database to reduce redundancy and improve data integrity.

1st Normal Form (1NF)

Purpose: Ensures that there are no duplicate columns in a table and that each column contains only atomic (indivisible) values.

Example:

-- Before 1NF CREATE TABLE Students ( StudentID INT, Name VARCHAR(100), Subjects VARCHAR(100) -- Multiple values in one field (Not atomic) ); -- After 1NF CREATE TABLE Students ( StudentID INT, Name VARCHAR(100) ); CREATE TABLE StudentSubjects ( StudentID INT, Subject VARCHAR(50) );

2nd Normal Form (2NF)

Purpose: Eliminates partial dependencies by ensuring that non-key attributes depend on the whole primary key.

Example:

-- Before 2NF CREATE TABLE Orders ( OrderID INT, ProductID INT, ProductName VARCHAR(100), Quantity INT ); -- After 2NF (Splitting into separate tables) CREATE TABLE Orders ( OrderID INT PRIMARY KEY ); CREATE TABLE OrderDetails ( OrderID INT, ProductID INT, Quantity INT ); CREATE TABLE Products ( ProductID INT PRIMARY KEY, ProductName VARCHAR(100) );

3rd Normal Form (3NF)

Purpose: Eliminates transitive dependencies by ensuring that non-key attributes depend only on the primary key.

Example:

-- Before 3NF CREATE TABLE Employees ( EmployeeID INT, Name VARCHAR(100), DepartmentID INT, DepartmentName VARCHAR(100) -- Transitive dependency ); -- After 3NF (Separate Departments table) CREATE TABLE Employees ( EmployeeID INT PRIMARY KEY, Name VARCHAR(100), DepartmentID INT ); CREATE TABLE Departments ( DepartmentID INT PRIMARY KEY, DepartmentName VARCHAR(100) );

6. Indexing & Performance Optimization

Optimizing database performance is crucial for efficient data retrieval and system scalability.

Indexes

Purpose: Speeds up searches by creating a structured way to access data quickly.

Example:

CREATE INDEX idx_users_name ON users (name);

Partitioning

Purpose: Splitting large tables into smaller, manageable partitions for improved performance.

Example:

CREATE TABLE orders ( order_id INT, order_date DATE, amount DECIMAL(10,2) ) PARTITION BY RANGE(order_date);

Sharding

Purpose: Distributing data across multiple servers to handle large-scale applications efficiently.

Example: In MongoDB, sharding can be achieved using:

sh.enableSharding("myDatabase"); sh.shardCollection("myDatabase.users", { "user_id": "hashed" });

Caching

Purpose: Using tools like Redis or Memcached to store frequently accessed data for faster retrieval.

Example: Storing a query result in Redis:

SET user:1001 "{'id':1001, 'name':'John'}";

7. Transactions & ACID Compliance (SQL Databases)

Transactions ensure that a series of operations are completed successfully and maintain database integrity. ACID properties define the reliability of database transactions.

Atomicity

Purpose: Ensures that all operations in a transaction are completed; otherwise, none of them are executed.

Example:

BEGIN TRANSACTION; UPDATE accounts SET balance = balance - 500 WHERE id = 1; UPDATE accounts SET balance = balance + 500 WHERE id = 2; COMMIT;

If one update fails, the entire transaction is rolled back, preventing partial updates.

Consistency

Purpose: Ensures that transactions transition the database from one valid state to another.

Example:

BEGIN TRANSACTION; UPDATE orders SET status = 'Shipped' WHERE id = 10; COMMIT;

Ensures that only valid updates are applied to the database.

Isolation

Purpose: Prevents transactions from interfering with each other, ensuring correct execution.

Example:

SET TRANSACTION ISOLATION LEVEL SERIALIZABLE; BEGIN TRANSACTION; SELECT * FROM accounts WHERE id = 1; UPDATE accounts SET balance = balance - 100 WHERE id = 1; COMMIT;

Ensures that another transaction cannot access the same data until the first transaction is completed.

Durability

Purpose: Ensures that committed transactions are permanently stored, even in case of a system failure.

Example:

COMMIT;

Once committed, the transaction changes persist even if the database crashes.

8. Database Security & Backup

Database security and backup strategies are essential to protect sensitive data and ensure data availability.

Authentication & Authorization

Purpose: Controls user access to the database using Role-Based Access Control (RBAC).

Example:

CREATE ROLE read_only; GRANT SELECT ON users TO read_only; GRANT read_only TO user1;

This example creates a read-only role and grants it to 'user1'.

Encryption

Purpose: Protects data both at rest and in transit.

Example: Enabling Transparent Data Encryption (TDE) in SQL Server:

CREATE DATABASE ENCRYPTION KEY WITH ALGORITHM = AES_256 ENCRYPTION BY SERVER CERTIFICATE MyCertificate;

This encrypts the database using AES-256 encryption.

Backups

Purpose: Prevents data loss by creating regular backups.

Example: Creating a full database backup in MySQL:

mysqldump -u root -p database_name > backup.sql

This command creates a backup of the specified database.

Replication

Purpose: Maintains multiple copies of the database to ensure failover and high availability.

Example: Setting up replication in MySQL:

CHANGE MASTER TO MASTER_HOST='master_host', MASTER_USER='replication_user', MASTER_PASSWORD='password', MASTER_LOG_FILE='mysql-bin.000001', MASTER_LOG_POS= 4; START SLAVE;

This configures replication on a slave server to sync with a master server.

9. Database Scaling & High Availability

Database scaling and high availability techniques ensure efficient performance, fault tolerance, and reliability.

Vertical Scaling

Purpose: Increases performance by adding more resources (CPU, RAM, Disk) to a single server.

Advantages: Simpler to implement, does not require data distribution.

Disadvantages: Hardware limitations, costly upgrades.

Example: Upgrading a database server from 16GB RAM to 32GB RAM.

Horizontal Scaling

Purpose: Distributes database workload across multiple servers to handle higher loads.

Techniques:

  • Sharding: Splitting large tables into smaller chunks distributed across multiple servers.
  • Replication: Creating copies of databases across multiple servers for redundancy.

Advantages: Better scalability, prevents a single point of failure.

Disadvantages: Complex implementation, requires data consistency management.

Example: A distributed database setup where different servers handle different portions of data.

Load Balancing

Purpose: Distributes database queries evenly across multiple nodes to optimize performance and prevent overload.

Techniques:

  • Round Robin: Directs queries to different servers in a circular order.
  • Least Connections: Routes queries to the server with the least active connections.
  • Geographic Load Balancing: Directs users to the nearest data center for faster response times.

Advantages: Improves response time, ensures even distribution of traffic.

Example: Using a load balancer to route queries dynamically to available database replicas.

High Availability Strategies

Purpose: Ensures continuous database availability and minimizes downtime.

Techniques:

  • Failover: Automatic switch to a backup database in case of failure.
  • Replication: Maintaining synchronized copies of a database across multiple locations.
  • Database Clustering: Grouping multiple database servers to work as a single system.

Example: A primary database failing over to a secondary database automatically during a system crash.
